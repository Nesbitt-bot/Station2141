## Serverless computing

Serverless is an approach to computing that:

- Offloads responsibility for common infrastructure management tasks to cloud providers and tools
- Enables engineers to focus their time and effort on the business logic specific to their applications or process

Serverless computing features:

- Strives for zero operational considerations
- Operations are managed by the cloud provider
- Developers can focus on writing applications

### Cloud provider responsibilities

- Cloud providers manage common labor-intensive infrastructure management tasks
    - Maximizing computer, memory and network utilization
    - Minimizing compute costs
- Other factors
    - Scaling
    - Low latency
    - High availability
    - Monitoring
    - Logging
    - Security

### Developer responsibilities

- Application development
    - Optimize their apps
    - Build cleaner code
    - Perform better testing
    - Add features
    - Improve user experience

### What's in a name

- There are servers in serverless computing
- Applications and code requires server space
- Serverless describes an end-user's experience
- The management needs of underlying servers are invisible to users

### Key attributes of serverless

- Requires no management or operation of infrastructures
- Enables developers to focus more narrowly on business logic
- Code runs on-demand on a per-request basis
- Scales transparently with the number of requests served
- End users pay only for the resources used
- Never pay for idle capacity

## Serverless Pros and Cons

### Pros

- No infrastructure requirements
- No servers to maintain
- Cloud providers ensure the reliability
- Fault-tolerance
- Enables developers to focus on code, not infrastructure
- Only pay for what you use
- Faster and more cost effective for many workload such as those that require parallel processing

### Cons

- Not ideal for workloads characterized by long-running processes
- Long-running workloads won't realize the benefits of the pay-per-use model
- A traditional server environment might be better for long-running workloads
- Serverless architectures aren't as portable as container and virtual machines
- Deeply integrating with managed cloud services can lead to vendor lock-in risks
- Often restart processes, known as a cold start
- For critical applications, even low-latency is unacceptable
- Monitoring and debugging are complex in any distributed system
- Microservices and serverless become more complex since requests and failures need to be traced among various services
- For example, you don't know whether an error originated in one microservice or another
- Not every language is necessarily supported in a serverless environment
- You cannot use serverless if the platform doesn't support your intended language

## Function as a service

- Function as a Service, known as FaaS, is an event-driven computing execution model
- Developers focus on building, running, and managing application packages as functions
- Functions are stateless, custom business code
- Application code is packaged as containers and runs in response to events or requests
- Manages server-side logic and state

### FaaS versus Serverless

- They are often conflated
- FaaS is a subset of serverless computing
- Many shared characteristics
    - Pay per use
    - No infrastructure management
    - Allow developers to focus on coding

### Function as a Service benefits

- Focus on code and not infrastructure
- Pay only for resources you use, when you use them
- Scale up or down automatically
- Achieve high availability

#### Functions in action

2.  Profile photo image is uploaded to Object Storage
3.  A Function is triggered
4.  An Function creates the profile photo thumbnail image
5.  Profile photo thumbnail image is uploaded to Object Storage

## Serverless Stack

- Serverless is a set of common attributes, not an explicit technology
- Many technologies are considered serverless
- FaaS is one example of a technology that is serverless and possesses these serverless attributes

### FaaS in the serverless stack

- Understood as the originating serverless technology
- Represents the core compute/processing engine
- Located in the center of most serverless architectures
- Functions perform custom logic and fire in response to events generated by other tools

### Serverless data layer

- Databased and storage are the foundation of serverless data layer
- Move away from provisioning instances with defined capacity
- Move toward models that scale linearly with demand

#### Traditional data layer

- Organizations provision databases on servers
- Capacity is limited by server size
- Cost is constant whether there is a lot or a little data

#### Object storage

- Internet communications data is largely unstructured
- Unstructured data doesn't conform to a relational database
- Storage architecture for handling large amounts of unstructured data
- Object storage provides scalability not offered by block and file storage

##### How object storage works

- Objects are discrete units of data
- There is no hierarchy as with directories
- Objects are self-contained repositories

### Streaming and messaging

- Serverless architectures are well-suited for event driven and stream-processing workloads
- Functions respond to events and messages communicated between microservices

### API gateways

- API gateways proxy web actions and provide HTTP method routing
- HTTP events often trigger functions
- API gateways expose those HTTP routes and provide additional functionality such as authentication

#### Elastic scaling

FaaS

- Each action is always instantly and inherently scaled

Other compute models

- Offer automatic, but slow, scaling that required careful tuning of auto-scaling rules

#### Capacity planning

FaaS

- Not required

Other computer models

- require a mix of automated scaling and capacity planning

#### Maintenance

- Providers manage all FaaS maintenance
- Containers and virtual machines require significant maintenance including update and management of operating systems, container images, connections, and others

#### High Availability (HA)

- FaaS is inherently highly available with no extra cost or effort
- Other compute models do provide high availability, but require cost and effort

#### Resource utilization

- Resources are never idle in FaaS
- All other models feature at lease some degree of idle capacity

#### Charging granularity and billing

- FaaS charges per block of 100 milliseconds
- Other models charge by the hour or perhaps by the minute

## Serverless Reference Architecture and Use Cases

### Serverless microservices

Microservices are

- The most common use case for serverless
- Focused on small services that do a single job and communicate via APIs
- Serverless functions
- Serverless can plug into architectures using other compute models such as PaaS and Containers

### Serverless use cases

- Data processing
- Massively parallel compute
- Stream processing

#### Data processing

- Serverless is well-suited to working with structured text, audio, image, and video data
    - Video transcoding
    - PDF processing
    - Thumbnail generation
    - Image object character recognition
- General process
    - The user needs to upload a profile photo
    - The profile photo image is uploaded to Object Storage
    - A function is triggered
    - An Function create the profile photo thumbnail image
    - The profile photo thumbnail image is uploaded to Object Storage

#### Massively parallel compute

- Is well-suited for serverless because each parallelizable task results in one action invocation

- MapReduce operations
- Web scaping
