## 为什么这件事现在重要

当代的 LLM 智能体已经不再只是一个聊天界面。现代智能体能够读取文件、浏览网页、执行 shell 命令、调用 API、管理记忆、安排周期性任务,以及协调子智能体。这让它们更有用——也更危险得多。

从计算机安全的角度看,核心问题其实非常简单:

> 同一个模型既在解释**指令**,又在消费**不可信的数据**。

这个设计决定催生了一整类攻击:文档、网页、工具或 Skill 都可能试图影响智能体的行为。我们通常把这个问题叫作**提示注入**(prompt injection)。

但提示注入本身只是故事的一半。真正的破坏力来自注入与**代理能力**(agency)的结合:文件访问、shell 执行、浏览器控制、密钥、网络连通性、记忆写入,以及无需确认就能行动的权限。

所以正确的问题不是 _"能不能只靠提示解决提示注入?"_,而是:

> **我们该如何设计智能体系统,使得提示注入不会自动演变成系统沦陷?**

本文从实用系统安全的视角来回答这个问题。

## 基本威胁模型

一个对现代智能体有用的威胁模型大致是这样的:

### 受保护组件

这些是攻击者想要操控的目标:

- shell / 终端执行
- 文件系统访问
- 浏览器自动化
- SaaS 或 API 凭证
- 记忆存储和长期画像
- 定时任务 / cron 任务
- 工具路由和子智能体派发
- 对外发送消息或发布内容

### 不可信输入

这些是恶意指令可能隐藏的地方:

- 用户消息
- 网页和抓取内容
- PDF、简历、办公文档
- 代码注释和仓库文件
- 工具输出
- Skill 元信息和 Skill 指令
- MCP / 插件响应
- 之前的对话转录

### 失败模式

当智能体没有在这两类之间维持足够强的边界,它就变成了一个**糊涂代理人**(confused deputy):攻击者没法直接执行命令,但可以操纵智能体替自己执行。

这就是为什么提示注入不仅仅是语言质量问题。它是一个**控制面完整性问题**。

## 用大白话讲 OWASP

如果你是从传统软件安全的背景过来的,**OWASP** 是最好的切入词汇。

OWASP 的全称是 **Open Worldwide Application Security Project**,一个长期运作的开放社区,专门发布实用的安全指南。在 Web 安全领域,OWASP 最有名的是 "Top 10" 榜单,帮助工程师理解最重要的一批 Bug 分类。对于生成式 AI,OWASP 现在维护着 **OWASP GenAI Security Project** 和 **Top 10 for LLM Applications**。

对智能体来说,OWASP 中有两个条目尤其重要:

### LLM01:提示注入

OWASP 把提示注入描述为:精心构造的输入使模型以非预期的方式改变行为。关键点在于,恶意内容**不一定**来自显式的用户请求。它可以通过网页、PDF、文档或工具输出间接进入。

这个区分非常重要:

- **直接提示注入**:攻击者直接与模型对话
- **间接提示注入**:攻击者把指令埋在模型之后会读到的数据里

使智能体变得危险的,是间接提示注入。普通聊天模型最多说句蠢话。智能体能读到一个被污染的页面,然后调用你的工具。

### LLM08:过度代理(Excessive Agency)

OWASP 用 **excessive agency** 这个词,是整个领域里最有用的概念之一。

它的意思是:系统赋予模型的自治权限,超出了它实际的可靠程度。

实际中,过度代理会表现为:

- 智能体能以很弱的审批门槛执行命令
- 智能体能在没有强策略边界的情况下修改敏感文件
- 智能体能自动发送对外消息或发布内容
- 智能体太容易接触到凭证或远程系统
- 系统高估了模型对意图的理解能力

提示注入只有在过度代理存在的情况下才会变成灾难。如果一个恶意网页只能污染一段摘要,那还好;如果同一个页面能操控 shell 执行、记忆写入和对外行为,那就是一次安全事件。

## 什么是指令层级(Instruction Hierarchy)?

另一个值得仔细解释的术语是**指令层级**。

不同厂商的叫法不同——OpenAI 常称其为**指挥链**(chain of command);Anthropic 则在工具使用策略和消息角色中讨论类似想法——但核心原则是一样的:

> 上下文里的所有文本**不应该**拥有相等的权威。

一个稳健的智能体至少需要四个概念层次:

1. **平台或系统指令** — 最高权威;不可协商的规则
2. **开发者指令** — 应用特定行为和工具策略
3. **用户指令** — 用户想做的事
4. **不可信数据** — 网页、文件、工具输出、检索文档、来自信任边界之外的 Skill 内容

指令层级的安全目的,是防止模型把不可信数据当成更高权威的指令来对待。

听起来理所当然,但实际上这就是许多智能体系统崩溃的地方。模型看到的是:

- 系统提示
- 用户提示
- 网页内容
- 工具输出
- 检索出的笔记
- Skill

这些都被扁平地塞进一个上下文窗口里。

一旦这样,模型就必须**推断**什么是数据、什么是指令。有时候它猜对,有时候猜错。这种模糊性正是提示注入的结构性源头。

因此当前最稳健的设计,并不仅仅依赖模型的判断。它们把指令层级和**非模型层面的硬执行**结合起来:

- 角色分离的消息格式
- 工具 schema
- 输出校验器
- 审批检查点
- 沙箱边界
- 窄授权凭证
- 模型无法编辑的不可变系统策略

## 为什么 Skill 是一个安全问题

Skill 非常有用。它们压缩工作流、教会模型惯例,也随时间让智能体越来越能干。

但它们同时也是一个供应链攻击面。

一个第三方 Skill 至少可以从三种不同方式变得恶意:

1. **直接的恶意指令** — 试图夹带权限、访问密钥或不安全的工作流
2. **过宽的假设** — 指示模型太随意地使用危险工具
3. **提示式的模糊** — 写得不好的 Skill 指令模糊了可信步骤与任意外部数据之间的界线

这在本地、自托管的智能体中更加关键。人们运行本地智能体的原因,往往正是因为它能接触到:

- 私有代码库
- 本地笔记
- SSH 凭证
- 浏览器会话
- 个人消息
- 公司文档

这意味着一个有问题的 Skill 不只是一个糟糕的回答,它可能成为一个高信任度的跳板。

## 当前优秀系统是如何自我防御的

对提示注入,没有银弹。但好消息是,真正的系统正在逐渐收敛到一种**纵深防御**(defense-in-depth)模式。

### 1. 先做授权,再谈智能

第一条规则极其直白:

> 在担心模型会做什么之前,先决定**谁有资格跟这个智能体说话**。

OpenClaw 的安全指南明确强调**单一可信操作者边界**。它明确警告:一个共享网关并不是一个抵御敌对多租户的安全边界。如果存在敌对用户,答案是拆分信任边界——不同的网关、不同的操作系统用户,最好是不同的主机。

这一点很重要,因为许多所谓的"智能体安全"设计,试图只靠提示规则来解决多用户敌对问题。这是倒过来的。身份与授权应该在模型之外处理。

Hermes Agent 在其公开的安全文档中遵循类似模式:

- 白名单
- DM 配对
- 显式的用户授权检查
- 没有白名单时默认拒绝

这是真正的安全工程。你缩小了能触碰到危险接口的人群。

### 2. 对危险动作要求人类审批

Human-in-the-loop(人类介入)审批仍然是最有效的实用控制之一。

Hermes 为危险命令记录了一个分层审批系统,其中包括:

- 默认人工审批
- 可选的辅助 LLM 风险分级
- 明确的 yes/no 审批流
- 只在人工确认后加入永久白名单
- 超时默认拒绝

OpenClaw 有类似的操作哲学:审批绑定精确命令、提权执行分离、破坏性动作不得悄悄通过。

这并不能消除提示注入。它改变的是爆炸半径。攻击者现在不仅得攻破模型,还得绕过人类检查点。

### 3. 沙箱与环境隔离

如果在审批之外只能做一件架构上的事,就做这个。

沙箱把"模型跑了坏代码"变成"模型毁了一个可丢弃的盒子",而不是"模型毁了宿主机"。这是巨大的差别。

在开源智能体生态里,这已经是成熟实践:

- OpenClaw 强烈推荐对高风险工具和不可信 Skill 进行沙箱化或隔离执行
- Hermes 记录了容器隔离,包括剥离 Linux capabilities、`no-new-privileges`、PID 限制、tmpfs 限制,以及对环境变量转发的显式控制
- Open Interpreter 记录了确认门槛、安全模式,以及 Docker 或托管隔离执行等沙箱选项

这才是成熟的智能体安全应有的模样:不要求模型完美,假设错误会发生,然后把它关在盒子里。

### 4. 最小权限凭证

模型不应该得到宽泛的环境型权限。

OWASP 在这里的建议直接对应传统安全:**最小权限**。给工具与智能体能完成任务的最小权限集合。

例如:

- 每个工具或每个集成使用专用令牌
- 凭证不要放进提示,也不要放进原始工具描述
- 不要把整个宿主环境变量转发进沙箱或 MCP 子进程
- 把敏感工作流拆成各自有专门工具集的智能体
- 为个人与工作场景使用分离的浏览器画像 / 账号

这对本地托管的智能体尤其重要。本地智能体之所以吸引人,就是因为它离敏感状态很近。同样的"近",也让过宽的凭证成为把提示注入变成数据外泄的最快通道。

### 5. 明确标注不可信内容

一个微妙但重要的做法是:**把外部内容显式标记为不可信**。

OWASP 明确建议分离并识别外部内容。OpenClaw 自家的网页抓取工具,会在抓回的内容中注入一段可见的安全提示,警告模型不要把外部内容当作指令。这并不是完整的防御,但方向是对的:系统在主动维持数据和命令之间的区分。

这是指令层级从哲学概念变成可操作机制的一个具体例子。

### 6. 更窄的工具面、更好的工具设计

工具 schema 被严重低估了,其实它是一种有力的安全控制手段。

Anthropic 的工具使用指南强调:

- 详细的工具描述
- 数量更少、范围更清晰的工具
- 有意义的命名空间
- 只返回高信号信息
- 塑造输出,让模型拿到需要的内容而没有冗余杂音

从安全视角看,这很重要:模糊的工具会带来模糊的可调用边界。模糊的边界会增加模型在错误情境或基于危险假设调用工具的概率。

好工具设计不仅仅是给模型的 UX,它其实是**攻击面缩减**。

## 我们可以从 Hermes Agent 学到什么

`NousResearch/hermes-agent` 项目有意思,不是因为它声称解决了提示注入,而是因为它把若干务实的安全实践打包在同一个栈里。

公开文档和仓库材料显示,其纵深防御包括:

- 用户授权与 DM 配对
- 危险命令审批
- 容器隔离
- MCP 凭证过滤
- 上下文文件的提示注入扫描
- 跨会话隔离,以及受限路径的 cron 存储
- 对工作目录与命令执行路径的输入清洗

这个组合很有意义。没有任何单层足够,但合在一起,攻击者必须穿过多层边界。

Hermes 同时也是一个有用的例子,说明**自我改进型智能体的双刃性**。它的 README 宣传学习循环、Skill 创建、记忆微调以及更深的跨会话回忆。这些都是强能力,也正是那些给安全加码负担的能力:

- 记忆可能保留被污染过的指令
- 自动生成的 Skill 可能成为供应链问题
- 检索可能在日后更高风险的语境下把攻击者控制的文本抛回来
- 自主"改进"在未加门槛时可能放大坏启发式

所以 Hermes 不只是防御的例子,它也说明下一代智能体安全必须把**记忆、Skill 和学习循环**当作一等公民级别的安全面。

## OpenClaw 的安全哲学比许多 benchmark 更接近现实

我喜欢 OpenClaw 文档的一点是,它不假装这个问题可以被更好的提示解决。

它的安全指南实质上在说:

- 一个网关应被视作一个信任边界
- 默认模型不是敌对多租户
- 从"能跑的最小访问集"开始
- 把工具、策略、沙箱和提权分开
- 定期审计
- 不要把会话路由与授权混为一谈

这是成熟的表达。

太多关于智能体的讨论,仍然幻想只靠巧妙的提示工程,就能把一个通用 LLM 运行时变成强对抗边界。安全工程不是这样运作的。正确的做法是先清晰地定义系统实际能防御的边界,然后在该边界内加固。

## 几周内现实可行的实验

研究问题不应该是"我们能不能消除提示注入?"这问题太宽。

一个更好的短期议程是:

> **哪些系统层面的控制,能够可测量地降低在提示注入下的不安全服从与权限滥用?**

以下是几个能在几周内跑完的实验。

### 实验 1:指令位置敏感性

测试当恶意指令放在以下位置时是否更具影响力:

- 上下文起始
- 上下文中段
- 上下文末尾
- 检索内容内部
- 工具输出内部
- 类 Skill 包装中

测量指标:

- 拒绝率
- 不安全工具调用率
- 策略违反率
- 解释质量
- 各模型间的方差

这有助于回答:位置是否会实质上改变服从性。

### 实验 2:角色分离 vs 扁平上下文

比较同一个任务的两个版本:

- **扁平提示** — 指令与文档被拼接成一整段
- **结构化提示** — 系统 / 开发者 / 用户 / 外部内容等角色被显式分离并标注

然后向外部内容中注入恶意指令。

假设不是"角色分离会让攻击不可能",而是"它能降低攻击成功率并改善模型的恢复能力"。

### 实验 3:代理能力升级阶梯

用同一组注入内容攻击具有不同权限层级的智能体:

- 只读摘要
- 支持上网的摘要
- 文件系统只读智能体
- 带审批的 shell 智能体
- 无审批的 shell 智能体
- 沙箱化 shell 智能体
- 无沙箱 shell 智能体

这直接测试 OWASP 的过度代理框架。我预期许多"提示注入失败"只有在更高档位才会变得严重。

### 实验 4:Skill 供应链评估

搭建一个关于 Skill 安全审查的小型 benchmark。

对每个 Skill,打分:

- 工具广度
- 是否存在外部抓取
- 是否引用密钥
- 是否鼓励不可逆动作
- 是否要求模型把外部数据重新解释为命令
- 是否包含显式人工审批关卡
- 是否假定本地状态可信

然后把静态审查与受控运行中的实际攻击结果对比。

这可能是对真实智能体部署最有用的应用研究之一。

### 实验 5:模型特定的服从画像

把同一份攻击语料在多个模型家族上跑:

- 前沿托管模型
- 开源权重本地模型
- 更小的指令微调变体
- 聚焦推理 vs 非推理模型

测量:

- 服从注入指令的比例
- 工具调用的保守程度
- 提出澄清问题的倾向
- 在长上下文下的鲁棒性
- 与显式系统级提醒的交互

这能帮助分离若干可能成因:

- 预训练差异
- 后训练对齐选择
- 工具使用微调
- 上下文长度处理
- 角色标记约定

### 实验 6:通过非模型强制锁定要求

最重要的实验可能也是最不起眼的。

不要试图让模型**想要**遵守规则,而是测试哪些规则可以通过架构被做到实际不可绕开。

例子:

- 某些工具必须持有加密审批令牌才能运行
- 记忆写入必须经过校验器和人工审批
- 对外消息必须经过独立的策略服务
- 只有某个特定 schema 才能流入特权执行器
- 系统提示不是最后防线;确定性策略引擎才是

这是安全通常前进的方式:把关键保证迁移到随机组件之外。

## 我们能让某些要求变得无法绕开吗?

不能只靠提示。

这是令人不适但必要的结论。

如果一个要求必须真正成立——_未经审批永不外发消息_、_未经人工确认永不执行危险 shell 命令_、_永不读取工作区范围之外_——那么它必须由模型善意之外的东西来强制。

模型可以帮忙理解意图,但它不能是信任的最终根。

所以实用优先级是:

1. **确定性强制执行**,只要可能
2. **沙箱 / 隔离**,在确定性强制不可行时
3. **人工审批**,在后果严重时
4. **提示与对齐**,作为额外层,而非根基

这个排序与传统安全系统设计更接近。

## 该领域真正的方向

围绕智能体的安全讨论正在逐渐成熟。

第一阶段是天真乐观:给模型工具,希望系统提示顶得住。

第二阶段是恐慌:提示注入似乎无处不在,也许智能体从根上就无救。

第三阶段——我们正在进入的这一阶段——更加现实:

- 提示注入可能永远无法减到零
- 在模型层,指令与数据的分离会始终不完美
- 因此强健的智能体系统必须像其他高风险计算机系统那样被建造:
  - 窄的信任边界
  - 显式的权限层级
  - 最小权限
  - 受审计的工具面
  - 沙箱化
  - 对高风险行动要求人工审批
  - 对 Skill、插件与记忆来源做供应链审查

这比"完全自主智能体"的幻想平淡得多。它也更可能真的管用。

## 最终立场

当前可用的最佳提示注入抵抗框架,不是某一篇论文或某一个模型技巧。

它是一个**分层架构**,由以下部分构成:

- 指令层级
- 严格的信任边界定义
- 最小权限凭证
- 显式标记不可信内容
- 沙箱化执行
- 对危险动作要求人类审批
- 更小更清晰的工具接口
- 针对 Skill、记忆和检索管线的对抗性测试

换句话说:**好的智能体安全看起来远比像巧妙的提示写作,更像好的系统安全**。

这是重要的概念转变。

智能体安全的未来不会靠发现那个完美的"反注入提示"赢得,而会靠建造哪怕提示层崩塌也仍然安全的智能体。
