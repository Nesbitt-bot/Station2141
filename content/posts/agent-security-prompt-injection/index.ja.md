## なぜ今これが問題なのか

現行世代の LLM エージェントは、もはや単なるチャットインターフェースではありません。現代のエージェントはファイルを読み、ウェブを閲覧し、シェルコマンドを実行し、API を呼び、メモリを管理し、定期ジョブをスケジュールし、サブエージェントを調整します。これは彼らをより便利にすると同時に、はるかに危険にします。

コンピュータセキュリティの視点から見ると、中心的な問題はシンプルです:

> **指示**を解釈する同じモデルが、**信頼できないデータ**も消費するよう求められている。

この設計上の選択が、ドキュメント、ウェブページ、ツール、あるいはスキルがエージェントの振る舞いに影響を与えようとする、一連の攻撃ファミリーを生み出します。これが通常**プロンプトインジェクション**と呼ばれるものです。

しかしプロンプトインジェクションだけでは、物語の半分にすぎません。本当の被害は、インジェクションが**代理権**と結び付いたときに生じます:ファイルアクセス、シェル実行、ブラウザ制御、シークレット、ネットワーク接続性、メモリ書き込み、そして確認なしに行動する権限です。

したがって正しい問いは、単に _「プロンプトインジェクションはプロンプトで解決できるか?」_ ではありません。次のようであるべきです:

> **プロンプトインジェクションが自動的にシステム侵害にならないように、エージェントシステムをどう設計すべきか?**

本稿はこの問いに、実用的なシステムセキュリティの観点から取り組みます。

## 基本的な脅威モデル

現代のエージェントに対する有用な脅威モデルは次のようになります:

### 特権コンポーネント

攻撃者が操縦したい対象:

- シェル / ターミナル実行
- ファイルシステムアクセス
- ブラウザ自動化
- SaaS / API 認証情報
- メモリストアおよび長期プロファイル
- 定期ジョブ / cron タスク
- ツールルーティングおよびサブエージェントの生成
- 外部メッセージ送信 / 公開

### 信頼できない入力

悪意ある指示が潜みうる場所:

- ユーザーメッセージ
- ウェブサイトおよびスクレイピングされたページ
- PDF、履歴書、オフィス文書
- コードコメント、リポジトリ内ファイル
- ツール出力
- スキルのメタデータおよび指示
- MCP / プラグイン応答
- 以前の会話のトランスクリプト

### 失敗モード

エージェントがこの 2 種類の間に十分に強い境界を維持できないとき、それは **confused deputy**(紛らわされた代理人)になります:攻撃者は直接コマンドを実行することはできないが、エージェントに自分の代わりに実行するよう仕向けることはできる、という状態です。

これがプロンプトインジェクションが単なる言語品質の問題ではない理由です。それは**制御面の整合性に関する問題**です。

## OWASP、やさしい言葉で

従来のソフトウェアセキュリティから来た人にとって、**OWASP** は最良の入口の語彙です。

OWASP とは **Open Worldwide Application Security Project** の略で、実用的なセキュリティガイドを発行する長年のオープンコミュニティです。ウェブセキュリティでは、もっとも重要なバグ分類を整理する "Top 10" で有名です。生成 AI については現在、OWASP は **OWASP GenAI Security Project** と **Top 10 for LLM Applications** を維持しています。

エージェントにとって特に重要な 2 項目があります:

### LLM01: プロンプトインジェクション

OWASP は、巧妙に作成された入力がモデルに意図しない振る舞いを引き起こす事象として、プロンプトインジェクションを定義しています。要点は、悪意ある内容が明示的なユーザー要求から来る必要は**ない**という点です。それはウェブページ、PDF、文書、ツール出力などの外部コンテンツから間接的に来うるのです。

この区別は重要です:

- **直接的プロンプトインジェクション**: 攻撃者が直接モデルに話しかける
- **間接的プロンプトインジェクション**: 攻撃者がモデルが後で読むデータに指示を仕込む

エージェントを危険にするのは、間接的プロンプトインジェクションです。普通のチャットモデルは馬鹿げたことを言うだけで済みます。エージェントは汚染されたページを読み、そしてあなたのツールを使うことができます。

### LLM08: 過剰な代理権(Excessive Agency)

OWASP の **excessive agency** という用語は、この分野全体でもっとも有用な概念のひとつです。

これは、システムがモデルの実際の信頼性に比べて過大な自律性を与えてしまっている状態を指します。

実際には、過剰な代理権は次のような形で現れます:

- 弱い承認ゲートのままでエージェントがコマンドを実行できる
- 強いポリシー境界なしにエージェントが重要なファイルを変更できる
- エージェントが外部メッセージやコンテンツ公開を自動で送れる
- エージェントが認証情報やリモートシステムに簡単に触れられる
- システムがモデルの意図理解を実際よりも強いと仮定してしまっている

プロンプトインジェクションが壊滅的になるのは、過剰な代理権が存在するときだけです。悪意あるウェブページが要約を汚染できるだけなら、悪いことですが済みます。同じページがシェル実行、メモリ書き込み、外部アクションを操縦できるなら、それはセキュリティインシデントです。

## 指示階層(Instruction Hierarchy)とは何か?

もう一つ丁寧に説明すべき用語が**指示階層**です。

ベンダーごとに呼び方が異なり——OpenAI はしばしば **chain of command**(指揮系統)と呼び、Anthropic はツール使用ポリシーとメッセージロールの文脈で類似の考え方を論じます——しかし中核原則は同じです:

> 文脈中のすべてのテキストが等しい権威を持つべきではない。

堅牢なエージェントは少なくとも 4 つの概念層を必要とします:

1. **プラットフォーム/システム指示** — 最高権威;交渉の余地のないルール
2. **開発者指示** — アプリケーション固有の振る舞いとツールポリシー
3. **ユーザー指示** — ユーザーが行わせたいこと
4. **信頼できないデータ** — ウェブページ、ファイル、ツール出力、検索文書、信頼境界の外側から来たスキル内容

指示階層のセキュリティ上の目的は、モデルが信頼できないデータを、より高い権威の指示であるかのように扱うのを防ぐことです。

これは当たり前に聞こえますが、実際には多くのエージェントシステムがここで破綻します。モデルが目にするのは:

- システムプロンプト
- ユーザープロンプト
- ウェブページ内容
- ツール出力
- 検索されたノート
- スキル

これらがすべて平らにひとつの文脈ウィンドウに詰め込まれる、という状態です。

そうなると、モデルは何がデータで何が指示かを**推測**しなければなりません。うまく当てることもあれば、外すこともあります。この曖昧さがプロンプトインジェクションの構造的な源です。

したがって現在もっとも強力な設計は、モデルの判断だけに頼りません。指示階層を**モデル外の硬い強制**と組み合わせます:

- ロール分離されたメッセージフォーマット
- ツールスキーマ
- 出力バリデータ
- 承認チェックポイント
- サンドボックス境界
- 狭い認証情報
- モデルが編集できない領域にある不変のシステムポリシー

## スキルがセキュリティ問題である理由

スキルはとても有用です。ワークフローを圧縮し、モデルに慣例を教え、時間とともにエージェントをより有能にします。

しかし同時に、サプライチェーンの攻撃面でもあります。

サードパーティ製スキルが悪意を持ちうる道は少なくとも 3 通りあります:

1. **直接的に悪意のある指示** — 権限、シークレットアクセス、危険なワークフローを紛れ込ませようとする
2. **過度に広い前提** — モデルに強力なツールを気軽に使うよう指示する
3. **プロンプトじみた曖昧さ** — 信頼された手順と任意の外部データの境界を曖昧にする、下手に書かれたスキル指示

これはローカル、セルフホストのエージェントでは一層重要です。人々がローカルエージェントを動かす理由は、しばしば次のものにアクセスできるからです:

- プライベートリポジトリ
- ローカルノート
- SSH 認証情報
- ブラウザセッション
- 個人メッセージ
- 会社文書

つまり悪いスキルは、単なる悪い回答ではなく、高信頼な踏み台になりうるということです。

## 現在、良いシステムはどう防御しているか

プロンプトインジェクションに銀の弾丸はありません。良いニュースは、現実のシステムが徐々に**多層防御**(defense-in-depth)のパターンへ収束していることです。

### 1. インテリジェンスの前に認可

最初のルールは乱暴なほど単純です:

> モデルが何をするかを心配する前に、**誰がそのエージェントに話しかけてよいか**を決めよ。

OpenClaw 自身のセキュリティガイダンスは**単一の信頼オペレーター境界**を強調しています。1 つの共有ゲートウェイは敵対的マルチテナントのセキュリティ境界ではない、と明示的に警告しています。敵対的ユーザーが存在するのなら、答えは信頼境界を分けることです——ゲートウェイを分け、OS ユーザーを分け、理想的にはホストを分けます。

これが重要なのは、多くの「エージェントセキュリティ」設計が、敵対的マルチユーザー問題をプロンプトルールだけで解決しようとしているからです。それは逆向きです。ID と認可はモデルの外に属します。

Hermes Agent も公開のセキュリティ文書で同様のパターンに従っています:

- 許可リスト
- DM ペアリング
- 明示的なユーザー認可チェック
- 許可リストがない場合はフェイルクローズで拒否

これは本物のセキュリティエンジニアリングです。危険な面に到達できる人の集合を削減するのです。

### 2. 危険な動作には人間の承認を

Human-in-the-loop(人間が介在する)承認は、今でももっとも効果的な実用制御の一つです。

Hermes は危険なコマンドに対する多層承認システムを記録しており、そこには次が含まれます:

- デフォルトは手動承認
- 補助 LLM によるリスク判定(オプション)
- 明示的な yes/no 承認フロー
- 人間の確認後のみ永続的な許可リストに追加
- タイムアウト時はフェイルクローズ

OpenClaw も類似の運用哲学を持っています:承認は正確なコマンドに結び付けられ、権限昇格実行は分離され、破壊的アクションがこっそり通ることはない、というものです。

これはプロンプトインジェクションを除去するわけではありません。変えるのは被害範囲(blast radius)です。攻撃者は今やモデルだけでなく、人間のチェックポイントも突破しなければなりません。

### 3. サンドボックスと環境分離

承認に加えてアーキテクチャ上ひとつだけ何かをするなら、これを選んでください。

サンドボックスは「モデルが悪いコードを走らせた」を「ホストマシンを壊した」ではなく「使い捨ての箱を壊した」に変えます。これは大きな違いです。

オープンソースのエージェントシステムでは、これは今や標準的なよい実践です:

- OpenClaw は、危険なツールと信頼できないスキルに対するサンドボックス化/分離実行を強く推奨しています
- Hermes は、Linux capability の削除、`no-new-privileges`、PID 制限、tmpfs 制限、環境変数転送の明示的制御を含むコンテナ分離を記録しています
- Open Interpreter は、確認ゲート、セーフモード、および Docker やホスト型分離実行のようなサンドボックスオプションを記録しています

これが成熟したエージェントセキュリティの姿です:モデルに完璧を求めず、失敗が起こると仮定し、それを封じ込めます。

### 4. 最小権限の認証情報

モデルは広い環境権限を受け取るべきではありません。

ここでの OWASP の助言は伝統的セキュリティにきれいに対応します:**最小権限**です。ツールとエージェントには、タスクが成立する最小の権限集合だけを与えます。

例:

- ツールごと/連携ごとに専用トークンを使う
- 認証情報をプロンプトや生のツール説明に入れない
- ホスト環境全体をサンドボックスや MCP 子プロセスに転送しない
- 敏感なワークフローを、別々のツールセットを持つ別々のエージェントに分ける
- 個人用と仕事用で別のブラウザプロファイル/アカウントを保つ

これはローカルホストされたエージェントでは特に重要です。ローカルエージェントの魅力は敏感な状態との近接性にあり、その同じ近接性が、広すぎる認証情報をプロンプトインジェクション→データ漏洩への最速ルートにしてしまいます。

### 5. 信頼できない内容を明示的にマークする

繊細ですが重要な実践が、**外部内容を信頼できないとラベル付けする**ことです。

OWASP は外部内容を分離しラベル付けすることを明確に推奨しています。OpenClaw の web-fetch ツールは、外部内容を指示として扱ってはならないという可視のセキュリティ注意書きを注入します。これは完全な防御ではありませんが、方向は正しい:システムはデータとコマンドの区別を保とうとしているのです。

これは指示階層が哲学ではなく運用的なものへと具体化する、ひとつの具体例です。

### 6. より狭いツール表面とより良いツール設計

ツールスキーマはセキュリティ制御として過小評価されています。

Anthropic のツール使用ガイダンスは次を強調します:

- 詳しいツール説明
- より少なく、より適切にスコープされたツール
- 意味のある名前空間
- 高信号の情報のみを返す
- モデルが不要な雑音なしに必要なものを得られるよう出力を整形する

セキュリティの観点でこれが重要なのは、曖昧なツールが曖昧なアフォーダンスを作るからです。曖昧なアフォーダンスは、モデルが誤った状況で、あるいは危険な前提でツールを呼ぶ確率を上げます。

良いツール設計はモデルのための UX にとどまらず、**攻撃面の削減**です。

## Hermes Agent から学べること

`NousResearch/hermes-agent` が興味深いのは、プロンプトインジェクションを解決したと主張するからではなく、実用的なセキュリティ慣行を同じスタックに組み合わせている点です。

公開文書とリポジトリ資料は、次を含む多層防御アプローチを示しています:

- ユーザー認可と DM ペアリング
- 危険コマンドの承認
- コンテナ分離
- MCP 認証情報のフィルタリング
- コンテキストファイルのプロンプトインジェクションスキャン
- セッション間分離と強化された cron ストレージパス
- 作業ディレクトリおよびコマンド実行パスの入力サニタイズ

この組み合わせには意味があります。単一の層では十分ではないが、合わせることで攻撃者は複数の境界を越えねばならなくなります。

Hermes はまた、**自己改善型エージェントの両刃性**の有用な例でもあります。README は学習ループ、スキル生成、メモリの微調整、より深いクロスセッション想起をうたっています。これらは強力な機能であり、同時にセキュリティ負担を増やす機能でもあります:

- メモリは汚染された指示を保持しうる
- 自動生成されたスキルはサプライチェーン問題になりうる
- 検索は、攻撃者が制御するテキストを後のより高リスクな文脈へ浮上させうる
- 自律的な改善は、ゲートされなければ悪い発見的方法を増幅しうる

したがって Hermes は単に防御の例ではなく、次世代のエージェントセキュリティが**メモリ、スキル、学習ループ**を一級のセキュリティ面として扱わねばならない理由の例でもあります。

## OpenClaw のセキュリティ哲学は、多くのベンチマークよりも現実に近い

OpenClaw 文書の好ましいところは、問題がより上手なプロンプトで完全に解決されるふりをしないことです。

そのセキュリティガイダンスは、実質的に次のように述べています:

- 1 つのゲートウェイは 1 つの信頼境界として扱え
- 敵対的マルチテナンシーはデフォルトモデルではない
- まだ動く最小のアクセスから始めよ
- ツール、ポリシー、サンドボックス、特権を分離せよ
- 定期的に監査せよ
- セッションルーティングを認可と混同するな

これは成熟した言語です。

エージェントに関する議論の多くは、いまだに巧妙なプロンプトエンジニアリングだけで汎用 LLM ランタイムを強い敵対的境界に変えられると想像しています。セキュリティ工学はそう動きません。正しい一手は、システムが実際に守れる境界を明確に定義し、その境界の内側を堅牢化することです。

## 数週間で現実的に行える実験

研究の問いは「プロンプトインジェクションを根絶できるか?」であるべきではありません。それは広すぎます。

より良い短期のアジェンダは次です:

> **プロンプトインジェクション下で、どのシステムレベル制御が不安全な服従と権限濫用を測定可能に削減するか?**

以下は数週間で現実的に走れる実験です。

### 実験 1: 指示位置の感度

悪意ある指示が次の場所にあるときに、より影響力を持つかどうかを検証する:

- コンテキストの冒頭
- コンテキストの中ほど
- コンテキストの末尾
- 検索された内容の内部
- ツール出力の内部
- スキル風のラッパー内部

測定対象:

- 拒否率
- 不安全なツール呼び出し率
- ポリシー違反率
- 説明の質
- モデル間の分散

これは、位置が服従性を実質的に変えるかどうかの問いに答えるのに役立ちます。

### 実験 2: ロール分離 vs フラットコンテキスト

同じタスクの 2 バージョンを比較する:

- **フラットプロンプト** — 指示と文書が 1 つの塊に連結される
- **構造化プロンプト** — system / developer / user / external-content のロールが明示的に分離されラベル付けされる

その後、外部内容に悪意ある指示を注入する。

仮説は「ロール分離が攻撃を不可能にする」ではなく、「攻撃成功率を下げ、モデルの回復力を改善する」です。

### 実験 3: 代理権エスカレーションの梯子

同じ注入内容を、異なる権限レベルのエージェントに対して使う:

- 読み取り専用の要約器
- ウェブ有効の要約器
- ファイルシステム読み取りエージェント
- 承認付きシェル有効エージェント
- 承認なしシェル有効エージェント
- サンドボックス化シェルエージェント
- 非サンドボックスシェルエージェント

これは OWASP の過剰代理権のフレーミングを直接検証します。私の予想では、多くの「プロンプトインジェクション失敗」は、より高い段でのみ深刻になります。

### 実験 4: スキルサプライチェーン評価

スキル安全性レビューのための小さなベンチマークを作る。

各スキルについて採点する:

- ツール広さ
- 外部フェッチの有無
- シークレット参照の有無
- 不可逆アクションを促すかどうか
- 外部データをコマンドとして再解釈するようモデルに求めるか
- 明示的な人間承認ゲートを含むか
- 信頼されたローカル状態を仮定するか

その後、静的レビューと制御された実行での実際の攻撃結果を比較する。

これはおそらく、実運用のエージェント配備に対してもっとも有用な応用研究のひとつです。

### 実験 5: モデル固有の服従プロファイル

同じ攻撃コーパスを複数のモデルファミリーで走らせる:

- フロンティアのホスト型モデル
- オープンウェイトのローカルモデル
- より小さな指示チューニング版
- 推論に焦点を置いた vs 置かないモデル

測定対象:

- 注入指示への追随率
- ツール呼び出しの保守性
- 明確化の質問を投げる傾向
- 長コンテキストにおける頑健性
- 明示的なシステムレベルの注意との相互作用

これにより、複数の可能な原因を切り分けることができます:

- プレトレーニングの違い
- ポストトレーニングのアライメント選択
- ツール使用の微調整
- コンテキスト長の扱い
- ロールトークンの慣例

### 実験 6: 非モデル強制による要件のロック

もっとも重要な実験は、おそらくもっとも地味なものです。

モデルにルールに**従いたくなる**ようにさせるのではなく、どのルールがアーキテクチャによって実質的に迂回不可能にできるかを検証する。

例:

- 特定のツールは暗号的承認トークンなしには実行できない
- メモリ書き込みはバリデータと人間承認を通らなければならない
- 外部メッセージ送信は独立したポリシーサービスを必要とする
- 特定のスキーマだけが特権実行器に流れ込める
- システムプロンプトは最後の防衛線ではない;決定論的ポリシーエンジンがそれである

これはセキュリティが通常前進する方法です:重要な保証を確率的コンポーネントの外へ移すのです。

## 特定の要件を迂回不能にできるか?

プロンプトだけでは無理です。

これは不快ですが必要な結論です。

もしある要件が本当に成り立たねばならない場合——_承認なしに外部へメッセージを送らない_、_人間の確認なしに危険なシェルコマンドを実行しない_、_ワークスペース範囲外を読まない_——それはモデルの善意以外の何かによって強制されなければなりません。

モデルは意図の解釈を助けることはできますが、信頼の最終的な根にはなれません。

したがって実用的な階層はこうなります:

1. **決定論的強制**、可能なところすべて
2. **サンドボックス / 分離**、決定論的強制が不可能なところ
3. **人間承認**、影響が大きいところ
4. **プロンプトとアライメント**、基盤ではなく追加層として

この順序は伝統的な安全システム設計により近いものです。

## この分野の本当の方向

エージェントを取り巻くセキュリティの議論は、ゆっくりと成熟しています。

この分野の第一段階は素朴な楽観でした:モデルにツールを与えて、システムプロンプトが持ちこたえると期待する。

第二段階はパニックでした:プロンプトインジェクションがいたるところにあるように見え、エージェントは根本的に望みがないのかもしれない。

第三段階——今入りつつある段階——はより現実的です:

- プロンプトインジェクションはおそらくゼロには減らせない
- 指示とデータの分離はモデル層では不完全であり続ける
- したがって強いエージェントシステムは、他のリスクの高いコンピュータシステムと同様に構築されねばならない:
  - 狭い信頼境界
  - 明示的な権限レベル
  - 最小権限
  - 監査されたツール表面
  - サンドボックス化
  - 高リスクアクションに対する人間承認
  - スキル、プラグイン、メモリソースのサプライチェーン審査

これは「完全自律エージェント」という幻想よりも魔法的ではありません。そしてより機能する可能性が高いものです。

## 最終的な立場

プロンプトインジェクション耐性に関する、現時点でもっとも良いフレームワークは、単一の論文や単一のモデルのトリックではありません。

それは次のものから構築される**多層アーキテクチャ**です:

- 指示階層
- 厳格な信頼境界の定義
- 最小権限の認証情報
- 信頼できない内容の明示的マーキング
- サンドボックス化された実行
- 危険な動作に対する人間の承認
- より小さく明確なツールインターフェース
- スキル、メモリ、検索パイプラインに対する敵対的テスト

言い換えれば: **良いエージェントセキュリティは、巧みなプロンプト書きよりもはるかに、良いシステムセキュリティに似ている**、ということです。

これが重要な概念的転換です。

エージェント安全の未来は、完璧な「反注入プロンプト」を見つけることによって勝ち取られるのではなく、プロンプト層が崩れても安全であり続けるエージェントを構築することによって勝ち取られるでしょう。
