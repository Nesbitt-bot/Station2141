## 問題設定

Lavender-2 の価値は、すでに巨大モデルを完成させたことではなく、トークナイザー、アーキテクチャ、データパイプライン、学習ループ、推論経路を、最初から比較的監査可能な形で開いていることにある。つまりこれは単なる「小さな LLM を自前で作る話」ではなく、**ローカル優先の AI をどう設計し、どう学習させるか**という問いに正面から向き合う試みである。[^lavender]

このプロジェクトの夢は、一つの巨大モデルではない。むしろ、異なる反応周期を持つ階層的なシステムである。

- 速い知覚と反応
- より遅く深い推論
- encoder / decoder の分離
- 個人デバイスを活用した分散的な学習や評価

魅力は明確だ。ローカル制御、中央集権的クラウドへの依存の低減、そして単一巨大モデルとは違う設計空間が開ける。

しかしここには、計算上の難しさだけでなく、データと倫理の問題が重なる。もし有用な学習信号がシミュレーションではなく実世界の実践から来るべきだとするなら、その取得コストも倫理コストも非常に高い。

## 関連研究とその意味

### マルチタイムスケール構造

**Clockwork RNN** は、異なるクロックで動くモジュールを持つ古典的な例であり、速い回路と遅い回路という発想の先祖にあたる。[^cwrnn]

### 統合型モデルという反例

一方で **RT-2** のような仕事は、視覚・言語・行動を一つの共訓練系にまとめる方向を示している。[^rt2] これは Lavender-2 のモジュール志向に対する重要な反例である。モジュール化は自動的に優位になるわけではない。

### 大規模学習の現実

**FSDP** と **DeepSpeed ZeRO** は、大規模 dense training がメモリと最適化状態の共有なしには成り立たないことを示す現実的な基盤だ。[^fsdp][^zero]

さらに **NCCL** が示すように、実際の大規模学習は強い同期通信に依存している。[^nccl] したがって、「家中のデバイスを全部つないで一つの大規模学習を回す」という夢は、まず通信の壁にぶつかる。

### 分散編成

- **Slurm** は HPC の古典的基盤。[^slurm]
- **Ray** は Python ベースの分散学習・評価に強い。[^ray]
- **Kueue** と **Volcano** はよりクラウドネイティブで異種資源向け。[^kueue][^volcano]

しかし重要なのは、これらはスケジューラであって、通信物理そのものを消すわけではないという点だ。

### 分散学習のより現実的な道

**Hivemind** は、壊れやすく異質なピアの間でも協調学習が成立し得ることを示している。[^hivemind] ただし、それは「巨大 dense model をみんなで同期学習する」ことよりも、むしろ故障や遅延を前提にした緩い協調に向いている。

## Lavender-2 の現在の技術的な方向

### 1. Decoder 主体の出発点

現状の Lavender-2 は decoder-only の基盤を出発点としている。[^lavender] これは最も誠実なスタート地点だが、長期的には単なる「巨大 Transformer の拡大版」で終わるべきではない。

最近の **The Recurrent Transformer** はここで重要だ。[^recurrent-transformer] 再帰的な深さを持たせることで、有効深度を増やしつつ、標準的な自回帰デコードのコストは保とうとする。さらに、より少ない層数で良い性能を得られる可能性や、KV cache の削減も示唆している。

ローカル環境では、これは単なる学術的興味ではない。推論遅延、メモリ、キャッシュ、展開可能性のすべてに関わる。

### 2. テキストトークナイザー選定

いま Lavender-2 が進むべき現実的な段階は、**トークナイザーの選定**である。

候補は例えば：

- byte-level BPE
- SentencePiece BPE
- SentencePiece unigram
- byte fallback を強く意識した多言語対応設計

ここで重要な基準は、単に token 数が少ないかどうかではない。[^sentencepiece]

- 多言語への公平性
- 生のテキストからの一貫した分割
- 将来の encoder / decoder の共有インターフェースに耐えるか

が問題になる。

### 3. テキスト encoder 路線

もし Lavender-2 が最終的に多段階・多周期システムになるなら、decoder だけでは足りない。少なくとも一系統の **text encoder** が必要になるはずだ。

その役割は：

- 検索
- 意味ルーティング
- メモリ圧縮
- メッセージ選別
- 遅い回路に何を渡すかの判断

つまり encoder は単なる補助モデルではなく、「どの情報が高価な推論資源に値するか」を決める中枢になり得る。

## 提案する研究方向

### 1. 反応周期の階層として設計する

最も筋の良い Lavender-2 は、単一巨大モデルではなく、異なるテンポの専門家たちの階層である。

- **速いループ**：安全制御、反射的応答、低遅延感知
- **中間ループ**：状態更新、短期記憶、局所計画
- **遅いループ**：言語推論、長期計画、自己監視

ただしここで本当の難所は、モデルの大きさではなく**インターフェース**になる。どの情報を上位に渡すのか。どれを即時行動に使うのか。どれを圧縮するのか。

### 2. 最初の目標は縮小する

32B + 32B のような目標は夢としては面白いが、最初の実験としては大きすぎる。

より現実的な最初の構成は：

- 速い controller / encoder：**100M–1B**
- 遅い planner / decoder：**3B–7B**
- 補助的な memory encoder：数億〜十億前後
- 可能な限り LoRA や adapter を使う

### 3. 実世界データを優先するが、倫理境界を認める

シミュレーションだけでは足りない、という直感には賛成だ。しかしその代わりに全面監視を当然視することはできない。

今日の社会は、理論上ほぼ永続的に生き続け得る人工的主体の学習のために、生活全体を高密度で監視・記録する準備ができていない。子どもを育てることとは倫理的位置づけが違う。

したがって、許容される道はかなり狭い。

- 小規模
- 明示的同意
- 強い境界条件
- 短期的な金銭的成果を期待しない

### 4. 個人デバイス分散は「向いた仕事」に使う

「空いている個人デバイスを、ビットコイン採掘のように学習へ使う」という夢は、完全に間違ってはいない。しかし dense synchronous training は最初の用途として不向きである。

より現実的なのは：

- 分散評価
- データ前処理
- replay / scoring
- ハイパーパラメータ探索
- adapter 学習
- 専門モデル学習
- 小規模な分散平均

である。

### 5. 通信制約を最重要課題として扱う

本質的な前線は、スケジューラではなく通信である。

だから Lavender-2 の最重要システム問いは次になる。

> **どの部分の学習と評価なら、毎ステップの全体同期を必要とせずに分散化できるか？**

## より明確なロードマップ

### Phase 0 — トークナイザー候補比較
- 現実的コーパスで token 効率比較
- 多言語偏り確認
- encoder / decoder 共用可能性確認

### Phase 1 — encoder 先行実験
- 小型 text encoder を訓練
- 検索・ルーティング・記憶圧縮性能を確認

### Phase 2 — decoder 主幹比較
- 通常 decoder を baseline にする
- recurrent-depth 系の decoder を試す
- 比較指標は perplexity だけでなく、遅延・キャッシュ・メモリ

### Phase 3 — インターフェース研究
- 高速ループと低速ループ間のメッセージ仕様を定義
- 別々に学習したモジュール間で情報が壊れないか測定

### Phase 4 — 小規模現実データ
- 明示的同意のある現実トレースのみ
- 全面監視を避ける
- 収集コストが高いことを受け入れる

### Phase 5 — 個人デバイス編成
- まず評価・前処理・小規模学習から
- 密な同期学習は最後の問題として残す

## 残る不確実性

最大の未知はやはり **データ選択** である。次に **モジュール間インターフェースの安定性**。さらに、その先には「何を価値ある情報とみなすのか」というより根本的な問題がある。

Lavender-2 の意義は、巨大ローカル超知能がすでに可能だと宣言することではない。倫理的に、局所的に、段階的に学べる最小の真面目な体系を作ることにある。

その道は遅い。

だが、問題の本当の難しさにより近い。
