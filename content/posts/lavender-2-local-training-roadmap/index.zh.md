## 问题是什么

Lavender-2 目前最有价值的地方，不是它已经训练出了什么超大模型，而是它把训练流程尽可能坦白地摊开了：分词器、架构、数据流水线、训练循环、推理路径，全都以一种可读、可审计、从零开始的方式摆在那里。它至少试图回答一个严肃问题：**如果我们真的想在本地做模型训练，我们到底需要面对什么？**[^lavender]

但 Lavender-2 真正有意思的，不只是“从零训练一个 decoder-only 模型”，而是更大的方向：能不能构建一种**本地优先、分层、多时钟、并且在伦理上还能站得住**的 AI 系统？

这里的梦想并不只是“小一点的聊天模型”。更接近的是：

- 快速感知与反应模块
- 更慢但更深的推理模块
- 可能分开的 encoder / decoder 路线
- 不同反应周期上的不同模型
- 最后，再让个人手上的闲置设备承担一部分训练、评估或数据处理任务

这个方向的吸引力很明显：

1. **本地控制** —— 不必把一切都交给中心化集群。
2. **架构诚实** —— 承认感知、动作、记忆、语言推理未必应该由一个模型以同样频率全部完成。
3. **伦理上的自觉** —— 不把海量监控和无限制数据抓取当成默认前提。

但问题也随之变得更硬：

- 真正的工程难点不只是规模，而是接口、路由、同步和通信。
- 真正的数据难点不只是量，而是**什么数据才值得学**。
- 真正的伦理难点则是：如果有效训练信号必须来自现实世界实践，那么今天的社会未必准备好了为此付出相应的监督与采集代价。

## 相关研究与影响

### 多时间尺度架构

最经典的近亲之一是 **Clockwork RNN**。它把模块分成不同的时钟频率，让快模块和慢模块处理不同时间粒度的信息。[^cwrnn] 这和 Lavender-2 想象中的“快回路 / 慢回路”非常接近。

这类工作提醒我们：现实中的智能系统并不需要所有部分都以相同节拍运作。

### 统一式共训练的反例

但也有相反方向的强反例。比如 **RT-2** 这样的工作，更接近把视觉、语言、动作整合进同一个共训练系统，而不是强行拆成多个彼此独立的模块。[^rt2]

这很重要，因为它构成了对 Lavender-2 模块化直觉的挑战：**模块化并不天然更优**。它可能更可解释，也更适合不同反应周期，但它同样会引入接口漂移和 credit assignment 问题。

### 大模型训练的现实基础设施

如果讨论真正的训练工程，就绕不开 **FSDP**、**DeepSpeed ZeRO** 和 **NCCL**。[^fsdp][^zero][^nccl] 这些技术之所以存在，是因为大规模 dense training 很快就会被参数、梯度、优化器状态和通信成本压垮。

这意味着：当我们说“把所有个人设备连起来训练模型”的时候，我们不是只在和 FLOPs 对抗，而是在和整个同步通信结构对抗。

### 编排系统前沿

在调度与编排层面，当前世界大致是这样的：

- **Slurm** 仍然是传统 HPC 的核心基线。[^slurm]
- **Ray** 在 Python 原生分布式训练/推理工作流中很强。[^ray]
- **Kueue** 与 **Volcano** 则代表了更云原生、面向异构资源与批处理工作负载的方向。[^kueue][^volcano]

但这些系统的意义，更多是“更聪明地安排工作”，而不是“消灭通信物理学”。

### 去中心化训练

**Hivemind** 证明了一件非常重要的事：在不可靠、异构、甚至跨互联网的节点之间做协作训练，并不是纯幻想。[^hivemind]

但它真正可贵的地方，不是证明“家里的所有设备都能轻松变成一个超大 dense trainer”，而是说明：如果更新规则本身为异构性和失败而设计，那么某种形式的协作学习确实可能成立。

## Lavender-2 当前更清晰的技术方向

我认为 Lavender-2 最好不要被理解成一个单一路径项目，而应该被理解成三条交织的技术路线。

### 1. Decoder 主干路线

现阶段的公开描述仍然以 decoder-only baseline 为核心。[^lavender] 这是合理的，因为它是最简单、最诚实、也最容易验证的一条起点。

但长期来看，我不认为答案只是“把普通 dense Transformer 做得更大”。

一个值得认真考虑的方向，是 **recurrent-depth** 类型的 decoder。最近的 **The Recurrent Transformer** 就很关键，因为它提出了一种方法：在保持标准自回归解码代价的前提下，提升有效深度，并在某些设置下用更少层数达到更好的交叉熵表现，同时还能减轻 KV cache 压力。[^recurrent-transformer]

这对本地系统尤其重要。因为本地系统要面对的，不只是参数规模，而是：

- 推理延迟
- 显存 / 内存压力
- KV cache 占用
- 更小设备上的部署可行性

所以，Lavender-2 在 decoder 路线上的问题，不应该只是“抄哪个更大的模型”，而应该是：

> **什么样的 decoder 架构最适合本地约束、未来模块化集成，以及对缓存和延迟敏感的环境？**

### 2. 文本分词器选择

这是当前真正该做、而且非常关键的一步。

Lavender-2 现在已经到了应该认真看待 tokenizer 的阶段。分词器不是“后处理细节”，它会深刻影响：

- 训练效率
- 多语言公平性
- 表达压缩率
- 后续 encoder / decoder 接口兼容性

当前最现实的候选大概包括：

- byte-level BPE
- SentencePiece BPE
- SentencePiece unigram
- 更强调 byte fallback 或多语言稳定性的方案

这里最重要的基线文献仍然是 **SentencePiece**。它的价值在于直接从原始文本训练子词模型，而不是把空格切词视为理所当然。[^sentencepiece]

对于 Lavender-2 来说，分词器选择至少要回答三个问题：

1. **多语言问题** —— 如果未来真的处理混合语料、代码、中文、英文等，英文友好但其他语言成本高的 tokenizer 是否值得接受？
2. **压缩和语义结构的平衡** —— token 更少不等于更好，切分方式是否保留了足够好的结构？
3. **未来模块接口** —— 如果后面会有 encoder 与 decoder 分工，那么 tokenizer 是否支持稳定的语义路由？

### 3. 文本 encoder 路线

这是我认为目前被低估但非常重要的一条线。

如果 Lavender-2 最终真要走向一个多模块、本地优先、不同反应周期的系统，那么 decoder-only 模型不太可能独自承担所有角色。至少会需要一条 **text encoder** 路线，用于：

- 语义路由
- 检索
- 记忆压缩
- 消息筛选
- 慢回路输入挑选
- 状态摘要

换句话说，encoder 不只是“小模型”。它可能是整个系统里决定“哪些东西值得交给更昂贵、更慢的模型处理”的那个筛子。

所以，一个更合理的阶段性路线可能是：

1. 先选 tokenizer
2. 再训练小型 text encoder
3. 验证 encoder 是否真的能改善检索、路由和记忆组织
4. 然后再考虑如何把它和更大的 decoder / planner 结合

## 提议的研究方向

### 1. 把系统看作不同反应周期的层级结构

最值得认真测试的 Lavender-2，不是“一个超级大模型做所有事”，而是：

- **快回路**：安全、反射、低延迟感知摘要
- **中回路**：状态跟踪、局部记忆更新、短时规划
- **慢回路**：语言推理、长期规划、解释、自我监控

这在工程上和认知直觉上都成立。但模块越多，真正的问题就越不是“模型够不够大”，而是：

> **这些回路之间的接口要怎么定义，才能在分别训练后仍然保持有用？**

### 2. 缩小第一阶段目标

双 32B 模型是一个有价值的梦想，但不是合适的第一里程碑。

更现实的路线是：

- 快速控制 / 感知或 encoder：**100M–1B**
- 慢速 planner / decoder：**3B–7B**
- 可选的记忆或语义 encoder：几亿到低十亿级
- 尽可能使用 LoRA / adapter，而不是一开始就做完全 dense retraining

这样做不是退缩，而是把问题从“纯粹被基础设施压死”拉回到“真正能测试架构”。

### 3. 现实数据优先，但承认伦理边界

我赞成你的直觉：真正有用的训练信号，最终还是要来自现实世界实践，而不仅仅是模拟环境。

但这里的伦理代价必须被正面承认。今天的社会并没有准备好为了训练一种理论上可长期存在、可复制、可重放的实体，而接受大规模持续监督和监控。我们对孩子的训练可以接受高投入，是因为孩子被当作独立个体，处于成长、依赖和有限生命的伦理框架中。人工系统并不自然继承这种位置。

所以，伦理上更可接受的路径只能是：

- 小规模
- 明确同意
- 强边界
- 不以短期金融回报为目标

### 4. 个人设备分布式，只做适合的任务

“像挖矿一样用所有闲置设备训练模型”的梦想并没有彻底错，但 dense synchronous training 不是第一落点。

更现实的用途是：

- 分布式评估
- 回放 / 重打分
- 数据清洗与归一化
- 超参数搜索
- adapter 训练
- 专家模型训练
- 小规模去中心化平均

这时，Hivemind 的意义才真正显现出来：它给出的是一种为异构和失败设计的协作学习思路，而不是一个把所有设备神奇变成统一集群的神话。

## 更明确的路线图

### Phase 0 — 基础工具清理
- tokenizer 候选实验
- 本地真实语料上的 token 效率与结构比较
- 多语言偏置评估

### Phase 1 — 先做 encoder
- 训练小型 text encoder
- 测试其在检索、路由、记忆压缩上的价值
- 把它视为系统级接口，而不只是 embedding benchmark

### Phase 2 — decoder 主干实验
- 保留普通 decoder baseline
- 研究 recurrent-depth / Recurrent Transformer 路线
- 比较的不只是困惑度，还包括延迟、KV cache、内存占用

### Phase 3 — 接口实验
- 定义快回路与慢回路之间的消息格式
- 测试压缩摘要是否保留任务关键语义
- 测量 separately trained modules 的接口漂移

### Phase 4 — 小规模现实数据实验
- 明确同意下的现实世界数据收集
- 拒绝把全面监控当成默认方案
- 接受这类数据昂贵、缓慢、难规模化

### Phase 5 — 个人设备分布式编排
- 先做评估、回放、数据处理、小模型训练
- 只在同步成本可接受处尝试去中心化平均
- 不把 dense 全局同步训练当成起点

## 仍然存在的不确定性

最大的未知仍然是 **数据选择**。

如果模拟不够真实，而全面监控又不伦理，那么真正可用的数据只会来自少量昂贵、边界清晰、明确同意的实验。那意味着进展会慢，也可能长期没有好看的短期商业回报。

第二个未知是 **接口稳定性**。模块化系统的优雅，常常死于接口漂移。

第三个未知是 **价值函数本身**。如果 Lavender-2 的真正方向，是寻找一种能改变行为、提高潜力、带来有效观察变化的信息结构，那么我们最终要问的可能不是“模型多大”，而是：

> 我们到底希望哪些变化被当作有价值的信息？

这比模型架构还更根本。
